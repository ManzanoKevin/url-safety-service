1. Scaling the URL list beyond memory

For an MVP, using in-memory storage is the right approach. It works for this case and fulfills the expected behavior, demonstrating the core logic. However, it does not scale. I would move the data to a NoSQL database like MongoDB, since the query is simple: is the URL malicious or not (true/false). I would also consider introducing Redis if needed, especially for frequently repeated URLs.

2. Scaling beyond a single system

From my personal experience, the service should sit behind a load balancer. Given its nature, I would configure resource usage with a defined minimum and maximum, and implement horizontal scaling. For example, maintaining a minimum of 3 pods and a maximum of 5 in production, depending on usage. If the service requires more capacity, it should be evaluated as a team.

Another region:
I believe the main issue here would be latency for users. I haven’t had direct experience working with multi-region setups beyond working with a DRP in nearby zones, but I would replicate the database in that region and focus on reducing latency for those users as much as possible, then evaluate the effectiveness of the solution.

3. URL update strategies

I’ve recently worked with queues, and I think they would be a good fit for this case since updates are constant and can be managed asynchronously. I would adapt the service to integrate with this approach, while keeping the main logic unchanged.

4. If I get woken up at 3am

Currently, I handle these situations as follows:
- I receive an alert about a service outage
- I review alerts (emails) that provide initial context, so I already have an idea of the issue before connecting
- While setting up my console environment, I check LENS to visualize the state of the pods and gather useful information depending on their status (running, crashloopbackoff, containerconfigerror, imagepullbackoff, etc.)
- I review logs from the affected pod to identify the root cause
- I evaluate and apply a solution to restore the service, ideally within 15 minutes
- The next day, I report the root cause and propose improvements to the service so the team can decide on next steps
- Postmortem

5. Would I change anything in the implementation?

Definitely, depending on the root cause. If logs are not clear enough, they should be improved. I would also enhance error handling. The response depends on the issue found, but the goal is to make the service more robust and observable. It could also be an infrastructure-related issue.

6. Application lifecycle

I would integrate CI/CD to support continuous improvement, implement API versioning, and design the system to scale progressively over time.

7. Deploying a new version

I would strictly follow the development lifecycle: start in DEVELOP, then move to QA to validate functionality, then PRE-PRODUCTION (if required), and finally PRODUCTION. I would ensure proper PR reviews so the team is aligned with what is being deployed. There should be a clear production release process, including rollback strategies in case of issues.

The service image should be built using Docker multi-stage builds, which helps surface potential code issues early and reduces the risk of unexpected failures in production.
